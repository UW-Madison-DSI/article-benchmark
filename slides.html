The validity problem

LLM-generated papers will be "LLM-shaped": cleaner notation, more predictable structure, and more consistent than real papers. Extraction scores on synthetic papers will be inflated vs. real-paper performance. The benchmark is useful for relative comparisons between pipeline versions, but absolute scores will not transfer directly. This is why we need a real-paper calibration set.

Key design decisions

Lessons from critically evaluating the initial plan

Initial plan

  • 300 papers, 5-week timeline
  • Scoring built in week 4
  • 6 real papers for calibration

Revised plan

  • 75 papers + perturbations, 8-week timeline
  • Scoring built first (weeks 1-2) -- now complete

  • Pilot in one domain, then expand to 6-8

Guiding principle: Validate the method before scaling it. Each phase has a gate that must pass before the next phase begins.

Phase 1: Real-paper calibration set

Ongoing, low priority  |  The work that makes the synthetic benchmark trustworthy

Directional validation (rank correlation) does not require large sample sizes. 10-12 papers covering 3-4 domains is enough to catch systematic blind spots.

Strategy

  • Keep existing 6 curated papers as the primary calibration set
  • Incrementally add 4-6 more to reach 10-12 total
  • LLM-assisted annotation: extract first, then correct errors
  • Prioritize papers from different domains than the existing 6
  • Partial annotation acceptable: equations + LaTeX + variables most important

Why this is no longer the critical path

LLM-assisted approach: Run extraction pipeline first, then have expert correct errors rather than annotate from scratch. 2-3 hours per paper instead of 40.

Validation question: "When the synthetic benchmark says version A is better than version B, is that true on real papers too?"

Difficulty tier design

Stratified difficulty ensures the benchmark discriminates across capability levels

EASY   20%   3 pilot papers

  • 1-2 equations, explicitly labeled
  • All variables defined inline near equations
  • Standard LaTeX, no notation tricks

MEDIUM   40%   5 pilot papers

  • 3-5 equations that reference each other
  • Some parameters in separate tables
  • Mix of display and inline equations

HARD   30%   4 pilot papers

  • 5-8 equations, nested subsystem models
  • Parameters scattered across sections
  • Symbol overloading, non-standard notation

ADVERSARIAL   10%   2 pilot papers

  • Equations in running prose (no display math)
  • Ambiguous notation, missing units
  • Distractor equations, inconsistent naming

Difficulty tiers are calibrated after generation: run the pipeline, verify Easy > Medium > Hard > Adversarial, reclassify outliers.

Scoring framework BUILT

Field-type-aware scoring following ExtractBench methodology

Three levels of measurement

Equation detection F1

Did we find the right number of equations? Precision (no hallucinated equations) and recall (no missed equations).

Field completeness

For each detected equation, what fraction of schema fields (name, LaTeX, variables, parameters, units, assumptions) are correct?

Aggregate breakdowns

Score by difficulty tier, by domain, by field type. Identify systematic weaknesses ("always gets LaTeX wrong" vs. "misses parameters in tables").

Phase 5: Validation

Weeks 7-8  |  Does this benchmark actually measure what we think it measures?

Validation checks

  • Correlation test: Plot synthetic vs. real scores. Target Pearson r > 0.6. If below 0.4, the benchmark needs rework.
  • Difficulty calibration: Verify score ordering Easy > Medium > Hard > Adversarial. Reclassify outliers.
  • Discriminative power: Run 2+ pipeline configs (different LLMs or prompts). Scores must differ. Same scores = benchmark lacks power.
  • Expert spot-check: 10-15 synthetic papers reviewed for scientific plausibility.

If validation fails

Low correlation (r < 0.4)
Increase Adversarial tier %. Add more "messy" generation instructions. Or invest in more real annotations instead of synthetic data.

Tiers do not separate
Re-examine generation prompts. The "Hard" and "Adversarial" instructions may not be producing genuinely harder extractions.

No discrimination
Add harder edge cases. Test with more diverse pipeline configurations.
