# Synthetic Benchmark Dataset Generation Plan

## Goal

Build a trustworthy extraction benchmark for the ag-model-database pipeline, starting with a validated proof-of-concept in one domain before scaling. The final v1 target is ~75 synthetic papers (~350 mechanistic equations) with a calibration set of 10-12 real annotated papers to spot-check that synthetic scores directionally predict real-world performance.

---

## Context

The ag-model-database project uses three LangGraph agents (Research, Model, Review) to extract mechanistic equations and ML model descriptions from dairy science papers into structured JSON artifacts. The current benchmark consists of 6 manually curated papers with hand-annotated ground truth in two formats:

- **Curated format**: Rich JSON with paper metadata, equations (LaTeX, variables, parameters, units, assumptions, source spans), and extraction metadata
- **DB format**: Slimmed JSON aligned to the Postgres schema (`papers`, `mechanistic_equation_candidates`, `ml_model_candidates` tables)

Manual curation is accurate but does not scale. We need a larger benchmark to detect 5-10% performance differences between pipeline versions and to cover the diversity of equation types, extraction difficulties, and edge cases.

---

## Approach: Ground-Truth-First Synthetic Generation

The core insight (from SynthIE, EMNLP 2023): LLMs are better at generating text FROM structured data than extracting structured data FROM text. We exploit this asymmetry by:

1. Defining the expected extraction output (ground truth JSON) first
2. Generating realistic paper text that contains exactly that information
3. Running the extraction pipeline against the synthetic paper
4. Scoring extraction output against the known ground truth

This guarantees perfect ground truth by construction.

### The Validity Problem

Synthetic papers generated by an LLM will be "LLM-shaped": cleaner structure, more consistent notation, and more predictable formatting than real papers. This means extraction performance on synthetic papers will likely be inflated compared to real papers. The benchmark is still useful for relative comparisons between pipeline versions, but absolute scores will not transfer directly.

To address this, we invest in a real-paper calibration set (Phase 1) and validate correlation between synthetic and real scores before scaling up synthetic generation.

---

## Phase 0: Scoring Foundation (COMPLETE)

Build the scoring framework first. If we cannot reliably score extraction output, nothing downstream matters.

### Why scoring comes first

LaTeX comparison is the hardest unsolved piece. Consider: `\frac{a}{b+c}` vs. `a / (b + c)` vs. `a(b+c)^{-1}`. These are semantically identical but string-different. SymPy can help with symbolic equivalence but chokes on domain-specific notation like `\text{DMI}`, subscript conventions, and unit annotations mixed into formulas.

We need to build and test the scoring function against the 6 existing curated papers before generating any synthetic data. If scoring doesn't work, the whole benchmark is compromised.

### Deliverables

1. **`scoring/latex_compare.py`**: LaTeX normalization and comparison
   - Whitespace/formatting normalization
   - SymPy-based symbolic equivalence (where parseable)
   - Fallback to token-level edit distance for unparseable expressions
   - Test against all 44 equations in the existing benchmark (compare curated ground truth to itself with perturbations to validate the scorer)

2. **`scoring/field_compare.py`**: Field-type-aware comparison functions
   - Identifiers: normalized string match
   - Numeric values: relative tolerance (1%)
   - Free text: embedding cosine similarity (with threshold tuning)
   - Arrays: set-based F1 with element-level matching
   - Nested objects: hierarchical weighted match

3. **`scoring/score.py`**: Paper-level and system-level metric aggregation
   - Equation detection F1 (did we find all equations?)
   - Field completeness per equation
   - Weighted overall extraction score
   - Per-domain and per-difficulty breakdowns

### Validation gate: Run the scorer on the 6 existing papers (extract with the current pipeline, score against curated ground truth). The scorer must produce sensible rankings before proceeding.

## Phase 1: Real-Paper Calibration Set (Ongoing, low-priority)

Keep the existing 6 curated papers as the primary calibration set. Incrementally add 4-6 more over time to reach 10-12 total, prioritizing domain diversity over volume.

### Why 10-12 is enough

The calibration set answers one question: "when the synthetic benchmark says version A is better than version B, is that true on real papers too?" Directional validation (rank correlation) does not require large sample sizes. 10-12 papers covering 3-4 domains is enough to catch systematic blind spots without becoming a bottleneck.

### Approach

- Use LLM-assisted annotation: run the extraction pipeline first, then correct errors rather than annotating from scratch. This is significantly faster than manual annotation.
- Prioritize papers from different domains than the existing 6 (which are mostly microbial growth and meta-analysis)
- Partial annotation is acceptable: equations + LaTeX + variables are most important; exhaustive parameter sourcing can be skipped
- This work happens in the background and does not block Phase 2 or Phase 3

### Time estimate

4-6 additional papers at 2-3 hours each (LLM-assisted) = 8-18 hours of expert time, spread across the project timeline.

### Deliverable: `benchmark/real_papers/` with 10-12 annotated papers in DB format.

## Phase 2: Pilot Domain - Microbial Growth (Weeks 3-5)

Build the template library and synthetic generation pipeline in a single domain before scaling. Microbial growth is the best pilot domain because the existing 6 benchmark papers are concentrated here.

### Template library (microbial growth only)

Build 15-20 equation templates covering:
- Baranyi primary growth model (with and without lag)
- Gompertz growth model
- Logistic growth model
- Gamma-type secondary models (temperature, pH, water activity)
- Cardinal parameter models
- Inactivation kinetics (log-linear, Weibull)
- Dose-response models

### How to build and validate

1. **Seed from existing benchmark**: The 6 curated papers contain 44 equations, mostly in this domain. Extract the structural patterns.
2. **Mine from references**: Predictive microbiology textbooks (Ratkowsky, Zwietering, Baranyi) have canonical forms.
3. **LLM-assisted drafting**: Prompt Claude with the schema and 2-3 examples to draft additional templates.
4. **Expert validation**: Every template must be reviewed for scientific accuracy. Budget 30-60 minutes per template. For 20 templates, that's 10-20 hours of expert time.

### Template structure (matches DB artifact schema)

Each template is a complete ground truth JSON containing:
- `equation_local_id`, `equation_name`, `equation_description`
- `equation_latex` (the formula)
- `variables[]` with name, unit, description
- `parameters[]` with symbol, value, unit, bounds
- `assumptions[]`
- `source_spans[]` with section, label_in_paper, text_context

### Generate pilot synthetic papers

- Generate 10-15 synthetic papers from the microbial growth templates
- Run the extraction pipeline on them
- Score against ground truth using the Phase 0 scoring framework
- Compare scores to the real-paper scores from Phase 1

### Validation gate: Synthetic scores should rank equation types similarly to real-paper scores. If the pipeline struggles with gamma models on real papers, it should also struggle on synthetic gamma model papers. If there is no correlation, diagnose why before scaling.

### Deliverables:
- `templates/microbial_growth.json` (15-20 validated templates)
- `benchmark/synthetic/pilot/` (10-15 synthetic papers + ground truth)
- Correlation analysis between synthetic and real-paper scores

## Phase 3: Scale to Full Domain Coverage (Weeks 5-8)

Only proceed here if the Phase 2 pilot validates that synthetic scores are meaningful.

### Expand template library

Add 3-4 domains at a time, prioritized by relevance to the ag-model-database's target literature:

| Priority | Domain | Templates | Rationale |
|----------|--------|-----------|-----------|
| High | Methane emissions | 12-15 | Active research area, many papers in pipeline |
| High | Milk yield prediction | 12-15 | Core dairy science, diverse equation forms |
| High | Feed intake / DMI | 10-12 | Frequently co-occurs with other models |
| Medium | Rumen fermentation | 10-12 | Complex multi-equation systems |
| Medium | Nutrient digestion | 8-10 | First-order kinetics, well-defined |
| Medium | Heat stress | 6-8 | Simpler models, good for Easy tier |
| Lower | Reproduction, body condition, environment, disease, economics | 4-6 each | Lower priority, add in v2 if needed |

### Target: ~80 total equation templates across 6-8 domains

### Generate full synthetic dataset

Compose equation sets into papers following difficulty tiers:

| Tier | Description | Papers | % |
|------|-------------|--------|---|
| Easy | 1-2 equations, explicitly labeled, variables inline | 15 | 20% |
| Medium | 3-5 equations, standard presentation, some params in tables | 30 | 40% |
| Hard | 5-8 equations, nested/referential, parameters across sections | 22 | 30% |
| Adversarial | Equations in prose, ambiguous notation, shared variables | 8 | 10% |
| **Total** | | **75** | |

### Generation pipeline

```
equation_templates[] + paper_metadata -> LLM prompt -> synthetic paper text -> validation
```

The LLM receives:
- 1-8 equation templates (the ground truth)
- Paper metadata (title, domain, species)
- Difficulty-tier-specific instructions (e.g., for Adversarial: "embed equations in running prose without display math, use inconsistent notation")

The LLM generates:
- A 2,000-5,000 word paper section
- Equations embedded naturally in prose
- Variable and parameter definitions woven into text

### Output format

Start with JSON text (paper text as a string field). PDF rendering is deferred to v2; it adds complexity (LaTeX toolchain, layout issues) without testing extraction logic.

### Paper metadata

- Synthetic DOIs: `10.0000/synthetic-benchmark-XXX`
- LLM-generated titles matching equation content
- Journal names sampled from real dairy science journals
- Keywords and domains from controlled vocabularies
- No real paper content copied verbatim

### Cost estimate

75 papers at ~3K tokens output each = ~225K output tokens for generation, plus ~500K input tokens for prompts with templates. At Claude Sonnet rates: roughly $5-15 per full dataset generation. Budget for 3-5 iterations during development: ~$50-75 total.

## Phase 4: Perturbation Augmentation (Week 8-9)

Expand coverage by systematically modifying Medium and Hard tier items.

### Perturbation types

1. **Variable renaming**: DMI -> FI, BW -> LW (tests synonym handling)
2. **Parameter value changes**: shift within realistic ranges
3. **Unit conversion**: kg -> lb, MJ -> kcal
4. **Notation changes**: T_min vs. T_minimum, Greek vs. Latin symbols
5. **Context shuffling**: same equations in different narrative framing

Deferred (too complex for v1):
- Equation form changes (expand/factorize) -- requires symbolic manipulation that may introduce errors
- Addition/removal of equations -- changes the ground truth structure, harder to track

### Target: 1-2 perturbation variants per Medium/Hard item = ~50 additional items, bringing total to ~125 synthetic papers.

---

## Phase 5: Validation and Calibration (Week 9-10)

### Validate synthetic benchmark quality

1. **Expert spot-check**: Domain expert reviews 10-15 synthetic papers for scientific plausibility. Not a full annotation pass; just "does this read like a real methods section?"
2. **Correlation analysis**: Plot synthetic-benchmark scores vs. real-paper scores for the extraction pipeline. Compute Pearson/Spearman correlation. Target: r > 0.6 for the benchmark to be considered useful.
3. **Difficulty calibration**: Verify Easy > Medium > Hard > Adversarial in extraction scores. Reclassify items that don't fit.
4. **Discriminative power test**: Run two different pipeline configurations (e.g., different LLM backends or prompt versions). Verify the benchmark produces different scores. If all configurations score the same, the benchmark lacks discriminative power.

### If validation fails

- If correlation with real papers is weak (r < 0.4): increase the Adversarial tier percentage, add more "messy" generation instructions, or invest in more real annotations instead
- If difficulty tiers don't separate: re-examine the generation prompts for each tier
- If the benchmark doesn't discriminate: add harder edge cases or more diverse equation types

## Deferred to v2

These items are explicitly out of scope for v1:

| Item | Reason for Deferral |
|------|-------------------|
| ML model benchmark items | The extraction pipeline has zero ML model examples; build the capability first, then benchmark it |
| PDF rendering | Adds LaTeX toolchain complexity; test extraction logic with JSON text first |
| Full 12-domain coverage | Start with 6-8 domains; add the rest after validating the approach |
| Scale to 300+ papers | Validate at 75-125 papers first; scale is cheap once the pipeline works |
| Perturbation via equation form changes | Requires robust symbolic manipulation; risk of introducing ground truth errors |

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| LaTeX scoring doesn't work reliably | ~~High~~ Resolved | Critical | **RESOLVED.** 3-tier comparison (normalize, SymPy, token fallback) works; 81 tests pass, all 44 equations self-score 1.0 |
| Expert time not available for real annotations | Medium | Medium | Reduced to 4-6 incremental papers (LLM-assisted); existing 6 papers are sufficient for initial calibration |
| Synthetic scores don't correlate with real scores | Medium | Critical | Pilot in one domain first (Phase 2); increase Adversarial tier; invest in more real annotations |
| Template library has scientific errors | Medium | High | Expert validation required; start with well-known canonical equations from textbooks |
| Benchmark doesn't discriminate between pipelines | Medium | Medium | Include adversarial cases; test with 2+ pipeline configs during validation |
| 10-week timeline slips | High | Low | Phases are independent milestones; partial completion (e.g., 50 synthetic papers) is still useful |
| LLM-generated papers are too formulaic | Medium | Medium | Study real paper structures; include specific instructions for variation in narrative style, section organization, and notation |
